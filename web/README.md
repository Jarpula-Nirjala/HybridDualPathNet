# HybridDualPatNet — Brain MRI Disease Classifier

Django web application for classifying brain MRI scans into **Alzheimer's Disease (AD)**, **Parkinson's Disease (PD)**, or **Healthy Control**, using a Hybrid Dual-Path model (EfficientNetV2-S + Swin-Tiny fusion) with **Grad-CAM++** explainability.

> **Not for clinical use.** Research and educational purposes only.

## Tech Stack

| Layer | Technology |
|-------|------------|
| Backend | Django 5.x, Django REST Framework |
| Database | PostgreSQL (prod), SQLite (dev) |
| Task queue | Celery + Redis |
| ML | PyTorch (CPU), timm, grad-cam, albumentations |
| Frontend | Django templates, Bootstrap 5, HTMX, Chart.js |
| Static files | WhiteNoise |
| Media (prod) | S3 / Cloudflare R2 via django-storages |
| Server | Gunicorn |

## Local Setup

### 1. Prerequisites

- Python 3.11+
- Redis (for Celery worker)
- Your trained `final_model.pth` checkpoint

### 2. Create virtual environment

```bash
cd neurofusenet_web
python -m venv .venv

# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate
```

### 3. Install dependencies (CPU PyTorch)

```bash
pip install -r requirements.txt --extra-index-url https://download.pytorch.org/whl/cpu
```

> **Why the extra index?** The CPU-only PyTorch wheel is ~200 MB vs ~2 GB for CUDA builds. Always use the `+cpu` index on deployment hosts without a GPU.

### 4. Environment variables

```bash
cp .env.example .env
```

Edit `.env`:

```env
SECRET_KEY=your-local-secret-key
DEBUG=True
DJANGO_SETTINGS_MODULE=config.settings.dev
REDIS_URL=redis://localhost:6379/0
MODEL_WEIGHTS_PATH=weights/final_model.pth
```

Copy your trained checkpoint:

```bash
mkdir -p weights
cp /path/to/final_model.pth weights/final_model.pth
```

### 5. Database & superuser

```bash
python manage.py migrate
python manage.py createsuperuser
```

### 6. Run services (3 terminals)

**Terminal 1 — Redis** (if not already running):

```bash
redis-server
```

**Terminal 2 — Django**:

```bash
python manage.py runserver
```

**Terminal 3 — Celery worker**:

```bash
celery -A config worker --loglevel=info --concurrency=2
```

Open http://127.0.0.1:8000 — sign up, upload an MRI, watch HTMX polling, view results.

### Testing without Redis

Set in `.env`:

```env
CELERY_TASK_ALWAYS_EAGER=True
```

Tasks run synchronously in the web process (useful for quick local testing).

## Plugging In Your Model

Your architecture from the training notebook is already in `inference/architecture.py` (`HybridDualPathModel` + `CrossFusionAttention`).

If you modified the architecture after training, update that file so `state_dict` keys match your checkpoint. The loader expects either:

- A dict with `model_state_dict` key (as saved in the notebook), or
- A raw state dict.

## Deploy to Render

### Provision order

1. **PostgreSQL** database (`neurofusenet-db`)
2. **Redis** instance (`neurofusenet-redis`)
3. **Web service** (`neurofusenet-web`)
4. **Worker service** (`neurofusenet-worker`)

### Steps

1. Push this repo to GitHub.
2. In Render Dashboard → **New** → **Blueprint** → connect repo → select `render.yaml`.
3. Set sync=false env vars when prompted (see table below).
4. After first deploy, run migrations if not auto-run:

   ```bash
   python manage.py migrate
   ```

   (Add a Render **pre-deploy** or **release** command: `python manage.py migrate`)

5. Create a staff user via Render shell:

   ```bash
   python manage.py createsuperuser
   ```

### Render environment variables

| Variable | Required | Notes |
|----------|----------|-------|
| `SECRET_KEY` | Yes | Auto-generated by blueprint |
| `DJANGO_SETTINGS_MODULE` | Yes | `config.settings.prod` |
| `ALLOWED_HOSTS` | Yes | `your-app.onrender.com` |
| `CSRF_TRUSTED_ORIGINS` | Yes | `https://your-app.onrender.com` |
| `DATABASE_URL` | Yes | Auto-linked from Postgres |
| `REDIS_URL` | Yes | Auto-linked from Redis |
| `MODEL_WEIGHTS_URL` | Yes | HTTPS URL to your `.pth` file |
| `MODEL_WEIGHTS_PATH` | Yes | `weights/final_model.pth` |
| `USE_S3_MEDIA` | Yes | `True` (ephemeral filesystem) |
| `AWS_ACCESS_KEY_ID` | Yes | R2 or S3 access key |
| `AWS_SECRET_ACCESS_KEY` | Yes | R2 or S3 secret |
| `AWS_STORAGE_BUCKET_NAME` | Yes | Bucket name |
| `AWS_S3_ENDPOINT_URL` | Yes | e.g. `https://<account>.r2.cloudflarestorage.com` |
| `AWS_S3_REGION_NAME` | No | `auto` for R2 |

## Deploy to Railway

### Provision order

1. Create Railway project: `railway init`
2. Add **PostgreSQL** plugin → copies `DATABASE_URL`
3. Add **Redis** plugin → copies `REDIS_URL`
4. Deploy **web** service from repo
5. Add separate **worker** service (same repo, start command below)

### Web service settings

- **Build command:**

  ```bash
  pip install -r requirements.txt --extra-index-url https://download.pytorch.org/whl/cpu && python download_model.py && python manage.py collectstatic --noinput
  ```

- **Start command:**

  ```bash
  gunicorn config.wsgi:application --bind 0.0.0.0:$PORT
  ```

- **Release command:**

  ```bash
  python manage.py migrate
  ```

### Worker service

Create a second service in the same project:

- **Start command:**

  ```bash
  celery -A config worker --loglevel=info --concurrency=2
  ```

- Share the same env vars as the web service (especially `DATABASE_URL`, `REDIS_URL`, `SECRET_KEY`, `MODEL_WEIGHTS_*`, S3 vars).

### Railway environment variables

Same as Render table above. Set via Railway dashboard or `railway variables set KEY=value`.

```bash
railway up
```

## CPU Inference & Memory Gotchas

| Concern | Detail |
|---------|--------|
| **RAM** | PyTorch + timm + dual encoders need **~1.5–2.5 GB RAM** per worker after model load. Render/Railway **free tiers (512 MB)** will OOM. Use **Starter (512 MB–1 GB)** minimum; **Standard 2 GB+** recommended for the Celery worker. |
| **Inference time** | 7-pass TTA on CPU: **30–120 seconds** per scan depending on plan CPU. |
| **Worker concurrency** | Default `--concurrency=2` loads the model once per process (~2× RAM). Use `--concurrency=1` on smaller plans. |
| **Ephemeral disk** | Uploads lost on redeploy without S3/R2. Always set `USE_S3_MEDIA=True` in production. |
| **Build time** | First deploy downloads ~200 MB PyTorch CPU wheel + model weights. Allow 5–10 min. |
| **Cold starts** | Free web services spin down; first request + model load adds latency. Workers should stay warm on paid plans. |
